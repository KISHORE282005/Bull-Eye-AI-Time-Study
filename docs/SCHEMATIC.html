1End-to-End Workflow

Six stages, orchestrated by app.py inside a single guarded pipeline. Each stage below shows what happens, the deep mechanics, and the data in / data out. The progress bar advances 10 → 25 → 60 → 70 → 90 → 100%.

Stage 01 · Input
Video Upload
app.py
📹

Operator uploads a manufacturing video

The user drops an mp4 / avi / mov / mkv file (≤ 2 GB) into the Streamlit uploader. The file is written once to videos/ and previewed inline; a size guard rejects anything over the 2 GB limit.

Why it matters: the raw video is the single source of truth. Everything downstream is derived from it, so the app keeps the original on disk (idempotent save — it is not re-written if it already exists) and stores the path in st.session_state.video_path so a Streamlit rerun never loses it.
IN File (bytes) OUT videos/<name>.mp4
Stage 02 · Transfer
Upload to Gemini
gemini/uploader.py
📤

Push the video to the Gemini Files API

The local file is uploaded to Google's servers via client.files.upload(). Because large videos are transcoded asynchronously, the uploader polls the file handle every 5 seconds while its state is PROCESSING and only returns once the media is ACTIVE.

Deep detail: Gemini cannot reason over a video until it has finished server-side preparation. Returning too early would make the next step fail, so this blocking poll loop is the synchronisation point between the app and the model.
IN local path OUT Gemini file handle (ACTIVE)
Stage 03 · Intelligence
AI Analysis
analyzer.py · prompts.py
🤖

Gemini 2.5 Flash observes the process

The video handle plus TIME_STUDY_PROMPT are sent to generate_content(). The prompt casts the model as an Industrial Engineering Expert and constrains it with 12 rules: return JSON only, track only the main operator, ignore background people, and split the work into discrete processes with start_timestamp / end_timestamp in HH:MM:SS.sss.

Critical design choice — AI observes, code computes. Rule 11 explicitly forbids the model from calculating Duration, Op1–Op5, WT1–WT5, TOCT, NVA or R-NVA. The AI only reports what it saw and when. This keeps every number deterministic and auditable instead of trusting an LLM's arithmetic. The raw response is saved to output/gemini_response.txt for debugging.
IN video handle + prompt OUT JSON text (processes + timestamps)
Stage 04 · Validate
Parse JSON
gemini/parser.py
📑

Clean, parse and structurally validate

Markdown fences (```json) are stripped, the text is run through json.loads(), and the structure is checked: the activities array must exist and every activity must contain all six required fields — otherwise a ValueError names the offending activity.

Why a hard gate here: LLM output is untrusted text. Failing fast with a precise error ("Activity 3 is missing 'start_timestamp'") prevents malformed data from silently corrupting the calculations two steps later.
IN JSON text OUT validated Python dict
Stage 05 · Compute
Calculation Engine
utils/calculations.py
⚙️

The Industrial-Engineering time-study engine

calculate_time_study(data) transforms bare timestamps into a full time study: it computes each process duration, classifies the activity (Working / Waiting / Walking / Rework), routes time into per-operator columns (Op1–Op5 / WT1–WT5), derives TOCT / NVA / R-NVA per process, and finally aggregates the overall analysis (cycle, working, walking, idle, VA, NVA).

Deep detail: the engine is a five-stage, idempotent pipeline — every output column is reset to 0.0 before it is written, so re-running can never double-count. See §3 for the internal flow.
IN timestamps only OUT enriched activities + overall_analysis
Stage 06 · Deliver
Report & Dashboards
report.py · pages/*.py
📊

Persist reports and render the dashboards

save_report() writes the canonical output/time_study.json, a styled multi-sheet Excel workbook (branded header, auto-fit columns, frozen panes, an Overall Analysis sheet) and a CSV. The app then reruns; every dashboard page reads the same JSON through TimeStudyLoader and renders KPIs, a process timeline, lean-waste analysis, and download buttons.

Single source of truth: because all five pages read one JSON file, the UI always reflects the most recent analysis. TimeStudyLoader back-fills any missing keys with safe defaults so a partially-formed file never crashes a page.
IN enriched data OUT time_study.json · .xlsx · .csv · live dashboards

3Calculation Engine — Internal Flow

calculate_time_study(data) runs five ordered stages over the activities list. Each stage is idempotent (resets its columns before writing), so the engine is safe to re-run on already-processed data.

1
validate_activity() · timestamp_to_seconds → calculate_duration → classify_activity Parses HH:MM:SS.sss to seconds (bad input → 0, never raises). duration = round(end − start, 3), floored at 0. Assigns activity_type and ensures required fields + operator default exist.
▼
2
update_operator_columns() Working duration → op{N} (productive). Waiting / Walking / Rework → op_wt{N} (lost time). N = operator number, default Operator 1, so single-operator studies fill only op1 / op_wt1.
▼
3
calculate_process_metrics() Per process: TOCT = duration always; NVA = duration for non-Working; R-NVA = duration only for Rework. Guarantees the invariant R-NVA ≤ NVA ≤ TOCT.
▼
4
calculate_overall_analysis() Sums durations by type into the study totals: cycle time, working (= VA), walking, idle (waiting), and non-value-added (waiting + walking + rework).
▼
5
write back to data Replaces data["activities"], sets data["overall_analysis"] and data["total_processes"]. This dict is what save_report() persists.

4Activity Classification → Metrics

Case-insensitive substring match on process_operation, checked in order Working → Waiting → Walking → Rework (first match wins; no match defaults to Working). The chosen type drives every downstream metric.

🔧 Working

Value-Added
Assembly · Bolt Tightening · Fastening · Pick / Place Part · Loading · Unloading · Machine Operation · Inspection · Testing · Welding · Grinding · Painting · Material Handling
TOCT = dur · NVA = 0 · R-NVA = 0Counts toward VA and Op columns

⏳ Waiting

Idle waste
Waiting · Searching (Tool / Material) · Machine Delay · Material Delay · Talking · Idle
TOCT = dur · NVA = dur · R-NVA = 0Feeds operator_idle_time

🚶 Walking

Motion waste
Walking · Transportation · Move · Walking to Machine · Walking to Rack
TOCT = dur · NVA = dur · R-NVA = 0Feeds walking_time

🔁 Rework

Repeat waste
Rework · Repeat Inspection · Repeat Tightening · Repeat Assembly
TOCT = dur · NVA = dur · R-NVA = durOnly category that raises R-NVA

⚠️ Ordering caveat: because Working is tested first, "Repeat Inspection" matches Inspection and classifies as Working, not Rework. Reorder the checks (or use more specific keywords) if Rework detection for those phrases is required.

6Key Formulas & Invariants

Every metric is a bucketed sum of process durations — no hidden weighting, fully reproducible.

MetricDefinition
Durationround(to_sec(end) − to_sec(start), 3), minimum 0
TOCT (Total Observed Cycle Time)= duration (always) ⇒ Σ TOCT = cycle time
NVA (Non-Value-Added)= duration if type ∈ {Waiting, Walking, Rework}, else 0
R-NVA (Repeat NVA)= duration if type = Rework, else 0
Value-Added (VA)= Σ Working durations = operator_working_time
Cycle time= Σ of all durations
VA %VA / (VA + NVA) × 100  (0 when VA + NVA = 0)
InvariantR-NVA ≤ NVA ≤ TOCT = duration   — enforced by utils/validator.py

8Reliability & Design Principles

How the pipeline stays robust against LLM variance, large uploads, and API failure.

🧮
AI observes, code computesThe model never does arithmetic (prompt rule 11); all metrics are deterministic Python — testable and auditable.
🔁
Idempotent stagesEvery engine stage resets its output columns before writing, so re-processing never double-counts.
🛡️
Fail-fast parsingMissing fields raise a precise ValueError; malformed timestamps degrade to 0 rather than crashing.
🚦
Graceful API handlingGemini 503 (busy) and 429 (quota) surface friendly retry messages; other errors show a full trace.
📐
Enforced invariantsR-NVA ≤ NVA ≤ TOCT = duration is validated, giving lean analysis a consistent basis.
🗂️
Single source of truthOne time_study.json feeds every page; TimeStudyLoader back-fills defaults so no page crashes.
